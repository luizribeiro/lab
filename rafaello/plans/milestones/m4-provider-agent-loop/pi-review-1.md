## Blockers

1. **`rfl chat` orchestration never loads the lock or spawns the provider/tool plugins.** Evidence: the roadmap demo requires `rfl chat` against mock provider + `read-file` (milestones/README.md:57), and the headline test assumes a pre-materialised `rafaello.lock` with `rfl-mockprovider` + `rfl-readfile` installed (scope.md:1296-1304). But the live `run_chat` still constructs a `BrokerAcl` with `plugins: BTreeMap::new()` and `tool_routes: BTreeMap::new()` (crates/rafaello/src/lib.rs:154-157), then only wires a frontend/session/controller (crates/rafaello/src/lib.rs:160-165). Scope.md has AL/CR modules, but no concrete `rfl chat` step that reads `rafaello.lock`, validates/compiles it, constructs plugin ACLs, computes `SpawnPaths`, creates `PluginSupervisor`, spawns the active provider, and spawns/eager-loads the readfile tool. Risk #12 even admits m2/m3 have no lazy-spawn-on-publish (scope.md:1592-1608) while Out of scope claims the tool is spawned on first dispatch (scope.md:1469-1475). Concrete fix: add a dedicated CLI/orchestration section before AL: lock load + V3 validation + `broker_acl::compile`, `compile_plugin` for installed provider/tool entries, `PluginSupervisor` construction, eager provider spawn, explicit readfile eager spawn (or explicitly scope lazy-spawn and test it), shutdown/cleanup for all plugin children, and tests for missing/invalid lock/provider/tool entries.

The readfile manifest also uses `${PROJECT_ROOT}` (scope.md:1040-1042, 1085-1090), but m1's closed placeholder set is lowercase `${project}`, `${home}`, `${plugin}`, `${cache}`, `${state}` (m1 scope.md:244-248), and the live parser enforces only those names (capability_path_template.rs:17-55). Concrete fix: rewrite both fixture manifests in the actual m1 schema (`schema`, top-level `name/version/entry/rafaello`, valid `[load]` shape, `openrpc.json` sibling), use `${project}`, and add manifest+lock compile tests for both fixtures before any subprocess tests.

## High

1. **Provider env injection needs active-provider context the supervisor does not have.** Evidence: PS3 wants `RFL_PROVIDER_ACTIVE` set from `[session].provider_active` (scope.md:705-716), but `PluginSupervisor::spawn` only receives `&CompiledPlugin` and `&SpawnPaths` (supervisor.rs:310-314), and `CompiledPlugin`/`PluginAcl` carry provider_id but not session activeness. Concrete fix: either drop `RFL_PROVIDER_ACTIVE` for m4, pass active-provider state through a new spawn argument/plan field, or have `rfl chat` spawn only the active provider and make the env var unnecessary.

2. **The negative matrix under-covers the required `in_reply_to` table.** Evidence: the RFC requires `provider.<id>.tool_request`, `provider.<id>.assistant_message`, `plugin.<id>.tool_result`, `plugin.<a>.rpc_reply`, and `frontend.<id>.confirm_answer` policies (security RFC:1027-1036). The m4 negative matrix names tool_result missing, provider tool_request missing/stale, but not provider assistant_message missing/stale or rpc_reply stale (scope.md:1321-1336). Concrete fix: add named tests for provider assistant_message missing/empty/multiple/stale `in_reply_to`; keep m2 rpc_reply missing and add stale-id coverage if m4 adds in-flight maps for rpc_reply.

3. **Tool outside-grant test proves plugin path validation, not lockin denial.** Evidence: the roadmap negative explicitly says ŌĆ£tool requested outside its grant denied at lockinŌĆØ (milestones/README.md:57). Scope's readfile plugin canonicalizes and rejects path escapes itself before reading (scope.md:1075-1084), then the negative expects `{ok:false,error:"path denied"}` from readfile (scope.md:1358-1362). That can pass even if lockin is misconfigured. Concrete fix: include a malicious/bypass fixture or test mode that attempts `std::fs::read` outside the grant without the readfile plugin's ancestor check, and assert the sandbox denies it. Keep the readfile path-traversal test as an additional plugin-level negative.

4. **The mock-provider pattern parser is underspecified for punctuation and result correlation.** Evidence: the regex captures `\S+` (scope.md:982-987), so `what's in README.md?` captures `README.md?`; PR2 does not record the requested path in the outstanding-call state but later must output `Here's what's in <path>` on `core.session.tool_result` (scope.md:992-996). Concrete fix: pin path extraction (strip trailing punctuation or require quotes), store request_id ŌåÆ path in mockprovider state, and add tests for `README.md?`, missing files, and multibyte UTF-8 content.

5. **`BrokerError::ProviderIdMismatch` is used but not actually specified in B4.** Evidence: B5 says hard-error `BrokerError::ProviderIdMismatch` if arg and ACL diverge (scope.md:536-543), and Risk #9 repeats it (scope.md:1564-1570), but B4's error extension list omits that variant (scope.md:486-497). Concrete fix: either add the variant with fields and tests or simplify `register_provider(canonical, peer)` to read provider_id exclusively from ACL, eliminating the mismatch state.
