# mcpfit M0 ‚Äî foundation crate + macro-first ergonomic API

## Goal

Spin the hand-rolled MCP server example at `fittings/examples/mcp-server/` into
a standalone crate `mcpfit/` whose primary interface is a `#[tool]` proc macro,
backed by a public builder API. By the end of M0 the example becomes a
~30-line macro-first demo and the legacy hand-rolled implementation is gone.

## Settled design

**Crate placement.** Sibling workspace at `lab/mcpfit/`. Independent release
cadence from fittings; depends on fittings by path during development.

**Builder API (canonical engine).**

```rust
Server::new("my-server", env!("CARGO_PKG_VERSION"))
    .tool(Tool::new("add")
        .description("Adds two numbers")
        .input::<AddArgs>()
        .handler(|args: AddArgs, cx: Cx| async move { Ok(args.a + args.b) }))
    .run_entrypoint().await
```

**Macro form (primary, README-headlined).**

```rust
/// Adds two numbers.
#[tool]
async fn add(args: AddArgs) -> Result<f64> { Ok(args.a + args.b) }

#[tokio::main]
async fn main() -> Result<()> {
    Server::new("my-server", env!("CARGO_PKG_VERSION"))
        .tool(add::TOOL)
        .run_entrypoint().await
}
```

The macro transforms `fn name` into `mod name { pub const TOOL: ToolSpec = ...; }`.
`.tool(add::TOOL)` is the registration shape ‚Äî `.tool(add)` was rejected as too
magical to make unsurprising.

**Handler signatures.** Always `async`. Two accepted shapes:
- `async fn name(args: A) -> Result<R>`
- `async fn name(args: A, cx: Cx) -> Result<R>`

No-args = `args: ()`.

**Return-type rules (no magic, type-enforced).**
- `String / &str` ‚Üí text-only response.
- `f64 / i64 / u64 / bool / ...` (numeric primitives + bool) ‚Üí text-only via
  `Display`. This is what makes `Result<f64>` work uniformly in builder + macro.
- `Structured<T>` where `T: Serialize + JsonSchema + StructuredObject` ‚Üí
  structured-content response with default text from compact JSON. Override text
  via `.with_text("...")`.
- `ToolResponse` ‚Üí full control: multi-content blocks, `is_error: true`, etc.

`StructuredObject` is a *syntactic-only* marker trait: `#[derive(StructuredObject)]`
implements it for structs only. It does **not** validate the runtime schema shape
beyond what schemars produces.

**Cx (call context).** Wraps `fittings::ServiceContext`. Exposes:
- `cx.check_cancelled()? ` ‚Äî returns `McpfitError::Cancelled` if the call was
  cancelled. `?`-friendly.
- `cx.progress(p).total(t).message(m).emit()` ‚Äî builder-style progress
  notifications. No-op when no progress token was negotiated.

**Lifecycle gating (split, not uniform).**
- `tools/list`: lenient ‚Äî allowed after `initialize`, even before the
  `notifications/initialized` notification.
- `tools/call`: strict ‚Äî `InvalidRequest` until the session is `Running`.
- Client-callable `tools/register` (when enabled): strict ‚Äî requires `Running`.
  This reverses the legacy example's "register before initialize" posture.

**Schema input (mutually exclusive variants).**
- `.input::<T>()` ‚Äî derive from `JsonSchema`.
- `.input_schema(json!({...}))` ‚Äî hand-write the schema (typed handler still
  required; raw schema does NOT imply raw `Value` handler).
- `.input_with_schema::<T>(json!({...}))` ‚Äî derive Rust deserialization from
  `T` but use a hand-tuned schema (descriptions, examples, defaults).

**ToolContent** is `#[non_exhaustive]` from day one with `Text`, `Image`, and
`Resource` variants matching MCP spec.

**Errors split.**
- Handler returns `Err(FittingsError::...)` ‚Üí JSON-RPC error response
  (protocol-level: bad args, method not found, cancelled).
- Handler returns `Ok(ToolResponse::error("..."))` ‚Üí successful JSON-RPC
  response with `isError: true` (tool ran, produced a model-visible failure).
- `String / Structured<T>` shorthands always produce `isError: false`. Reach for
  `ToolResponse` to surface a tool-level failure.

**Runtime mutability.**
- Default: tool list is immutable after server construction.
- `Server::allow_runtime_registration()` exposes client-callable `tools/register`.
- `Server::allow_dynamic_tools()` (separate knob) advertises
  `tools.listChanged` without exposing client registration.
- Either knob auto-on advertises `tools.listChanged: true` in initialize response.

**Transport / entrypoint split.**
- `Server::serve_stdio()` ‚Äî just builds the fittings stdio transport, applies
  MCP cancellation config, serves. For embedders/tests.
- `Server::run_entrypoint()` ‚Äî does the FITTINGS env / `serve` first-arg /
  exit-code dance. No clap, no subcommands beyond `serve`.

**Cancellation wiring.** `notifications/cancelled` with `requestId`, applied by
the server helper. Same constants as the legacy example.

**Server info.** Required at `Server::new(name, version)` ‚Äî no hardcoded values.

## Commit guidelines (project-wide, non-negotiable)

- One commit = one idea. If "X and Y" describes it, split.
- Tests live in the same commit as the code they test. No "add foo" + "add tests
  for foo" pairs.
- Size limits: ‚â§ 3-5 files, ‚â§ 100 lines per commit. **Pure-deletion commits
  may exceed this.**
- Each commit must compile and pass tests on its own.
- Use the code-reviewer agent before each commit.

## Test invariants (every commit)

After every commit:

```
cargo test --manifest-path mcpfit/Cargo.toml
cargo test --manifest-path fittings/Cargo.toml -p mcp-server
```

Existing `fittings/examples/mcp-server` tests must remain green until they are
explicitly replaced by mcpfit-backed equivalents in Phase 9.
