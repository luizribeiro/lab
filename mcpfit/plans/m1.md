# mcpfit M1 — Client API

## Goal

Add `mcpfit::Client` so users can connect to MCP servers, not just expose them.
Lands a small upstream change to `fittings::Client` first, then builds the
typed MCP client on top.

By the end of M1 the existing `mcpfit/example/` crate gains a second binary
(`bin/client.rs`) that demonstrates the client, plus a default-running smoke
test that exercises the client against the example server.

## Settled design

### Slice A — fittings upstream

Lives in `fittings/crates/client/`. Two new generic primitives. Nothing
MCP-specific lands in fittings.

**Inbound notifications.** Add `Client::subscribe_notifications() ->
broadcast::Receiver<InboundNotification>`. Bounded broadcast (default 1024).
Multi-consumer. Slow listeners get `RecvError::Lagged(n)` — they don't block
the worker. Reimplement the existing `with_notification_handler` callback API
as a thin wrapper over `subscribe_notifications` so it stays as compat sugar
without being the primary path. Add a regression test that the worker reads
the transport continuously even when no calls are pending.

**Pending calls.** Add `Client::start_call(method, params) -> PendingCall`
(sync, not async — enqueue is fire-and-forget into the existing unbounded
mpsc; failures after enqueue surface via the pending result). Keep
`Client::call(method, params)` working — it becomes
`self.start_call(method, params).await`.

```rust
pub struct PendingCall {
    id: JsonRpcId,
    rx: oneshot::Receiver<Result<Value, FittingsError>>,
}

impl PendingCall {
    pub fn id(&self) -> &JsonRpcId;
    pub async fn result(self) -> Result<Value, FittingsError>;
}
impl IntoFuture for PendingCall { /* ... */ }
```

Opaque struct + `IntoFuture` so `pending.await` works without exposing a
`BoxFuture` field.

### Slice B — mcpfit::Client

Lives in `mcpfit/crates/mcpfit/src/client.rs`. Reuses the existing protocol
types in `mcpfit/crates/mcpfit/src/protocol.rs`.

**Connection / handshake.** Three explicit methods plus one ergonomic helper:

```rust
Client::connect(connector)               // auto-handshake, returns ready
Client::connect_uninitialized(connector) // no handshake
client.initialize() -> InitializeResult
client.initialized() -> Result<()>       // sends notifications/initialized
Client::spawn(cmd)                       // sugar for connect(SubprocessConnector::new(cmd))
```

Be careful: `SubprocessConnector::new(cmd)` already injects `FITTINGS=1` and
`.arg("serve")` (`fittings/crates/client/src/subprocess.rs:42`) — `Client::spawn`
must not double-add them.

**Tool calls.** Two layers. Plain calls take `args: impl Serialize` directly:

```rust
client.list_tools() -> Vec<ToolInfo>
client.call_tool_raw(name, args: impl Serialize) -> Result<ToolResponse, McpfitError>
client.call_tool(name, args: impl Serialize) -> Result<ToolResponse, McpfitError>
//   ^ errors on is_error: true via new McpfitError::ToolFailed(ToolResponse)
```

**Progress / cancellation.** Separate handle-style API for progress-enabled
calls. Plain `call_tool` future drop is local-only — no auto-cancel, since
that fires during timeout/select races and surprises people.

```rust
let mut call = client.call_tool_with_progress(name, args).start().await?;
while let Some(progress) = call.progress().recv().await { /* ... */ }
let response = call.await?;     // ToolResponse, errors on is_error
call.cancel().await?;           // sends notifications/cancelled with the request id
call.missed_progress_count();   // dropped progress events when buffer was full
```

Implementation: progress tokens are mcpfit-generated (allow caller override
later, not in M1). Token is injected into `tools/call.params._meta.progressToken`.
A background task subscribed to fittings notifications routes `notifications/progress`
to the per-call channel by token. Per-call channel is a bounded mpsc (capacity
64) using `try_send` + drop-on-full + a missed counter exposed on the handle —
one slow consumer must not back-pressure the global router.

**Raw notifications escape hatch.** `client.notifications() -> broadcast::Receiver<InboundNotification>`
delegates to the fittings broadcast, useful for `notifications/tools/list_changed`
and debugging.

**Routing task lifecycle.** mcpfit::Client owns a tokio task that subscribes
to fittings notifications and routes by token. Aborted on `Client` drop. When
the upstream stream closes, all active progress channels close cleanly.

### Out of scope for M1

- Server-initiated requests (`sampling/createMessage`, etc.) — M2.
- Reconnect / retry.
- Tool result schema validation.
- Typed result decoding (`call_tool_as<T>`).
- Configurable broadcast capacity at runtime (1024 default fine).

## Commit guidelines (unchanged from M0, non-negotiable)

- One commit = one idea.
- **Tests live in the same commit as the code they test.** Edge-case tests
  for already-landed code may be a follow-up commit *only* when inlining
  would be impossibly large.
- Soft size guideline: ≤ 3-5 files, ≤ 100 LOC. Some commits will exceed when
  impl + tests is one idea — that's fine. Pure-deletion commits exempt.
- Each commit must compile + tests pass on its own.
- Use the code-reviewer agent before each commit.

## Test invariants (every commit)

After every commit:

```
cargo test --manifest-path mcpfit/Cargo.toml --workspace
cargo test --manifest-path fittings/Cargo.toml --workspace
```

Existing M0 tests (mcpfit unit + UI compile-fail tests + the example's
`stdio_e2e.rs` integration suite) must remain green throughout.

   Adds `InboundNotification`, bounded broadcast plumbing, `Client::subscribe_notifications()`,

   notifications when no calls are pending, and slow subscribers observing
   `Lagged(n)` instead of blocking the worker.
   Files: `fittings/crates/client/src/lib.rs` — ~150 LOC.
